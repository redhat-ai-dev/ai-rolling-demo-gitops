| Engineering workflow captures a repeatable process on how to deliver functionality  | \- EPICs captures the logical grouping of stories and tasks and owned by a single Scrum Team (Team field) \- EPICs (without features) can be used to capture maintenance work or function updates. Documentation tasks (component=Documentation) can be created under the EPIC if needed.  ~~that does not impact the user (Docs)~~ \- Scrum teams (Team field) triage, refine and maintain their backlog \- (Scrum Leads) lead and drive agile ceremonies (Sprint Planning, Refinement and Retrospective) \- (Scrum leads) Coordinates work across teams  \- Scrum team plans Sprint: stories, tasks, bugs (feature work, interrupt work, customer issues) \- (Scrum Leads) Coordinate Sprint Reviews. Ensures demos are completed and shared \- (Scrum Leads) Leads Retrospective and captures improvement opportunities as tasks (Component: Continuous Improvement) or issues under specific component for planning into future Sprints \- Complete items in sprint as planned   Note: Needs to be updated to reflect blended QE and Eng teams | Epic Workflow **NEW Status:**  Assignee, Team, Priority, Component  **Planning  status:**  Description (including Acceptance Criteria), T-Shirt Size best guess on work size. Epic is refined and epics owner is working on defining/grooming stories/tasks **Todo status:**  All expected Stories/Tasks are defined within the Epic  **IN PROGRESS status:** Stories are in progress, version is updated **Dev Complete status:** all linked stories, spikes, and tasks are in closed status. At this stage, as engg and QE are working as a blended team so automation (E2E) is covered in the same engg epic thus even if the automation (E2E) ticket is not complete we can move the epic status to dev complete **RELEASE PENDING status:**  Release Notes field in Jira to be updated(Release Notes Text and Release Notes Type), Release is pending and all work (stories, spikes, automation etc) is done **CLOSED status:**  Release(GA) has happened (For Epics that are not included as part of a release, it can be closed after all acceptance criteria have been fulfilled). | [RHIDP Team Dashboard](https://issues.redhat.com/secure/Dashboard.jspa?selectPageId=12360794) [RHIDP Hygiene](https://issues.redhat.com/secure/Dashboard.jspa?selectPageId=12363716) [RHIDP Sprints](https://issues.redhat.com/secure/RapidBoard.jspa?rapidView=17761&projectKey=RHIDP&sprint=67658#) [RHIDP Triaging](https://issues.redhat.com/secure/Dashboard.jspa?selectPageId=12360094) |

| Maintenance process captures how we handle code that produces unexpected results | \- Captures unintended behaviour  \- Anyone can report Bugs in RHDHBugs jira project  \- Link to Support Case (RHDHSupp) if it exists  \- Does not need a Parent Feature or EPIC unless it’s related to an existing Feature  \- Triaged by Scrum team and prioritized into affected releases \- Backporting considerations based on [criteria](https://docs.google.com/document/d/1IjMH985f3XUhXl_6drfUKopLxTBoY0VMJ2Zpr_62K2g/edit?tab=t.0#heading=h.qvw4t7r6cmox) \- Engineering teams can decide when to backport straightforward changes, but must inform the Release Manager (RM). For complex situations or those requiring Product Management (PM) input, the RM should facilitate a discussion with relevant stakeholders. See RM [Backporting Process](https://docs.google.com/document/d/13OkypJ3u_7Jq6kEhKhjEFwHQ12oPFDKXVzFjYW4XLdk/edit?tab=t.0) | Bug Workflow **NEW Status:** Description, Acceptance criteria, Refinement, Story point **Refinement status**: Acceptance Criteria, Story Point, Team, Priority  **Backlog:**  Waiting to be assigned and included in a sprint **IN PROGRESS status:** The bug is in progress **Review status:** PR is up for review and has been tested with all the required tests. **Release Pending status**: Release Notes field in Jira to be updated(Release Notes Text and Release Notes Type), Release is pending, and work to be verified  **Closed status:**  PR has been merged and verified |  |
| Support Engagement captures how we work with support engineers to address  customer found issues | \- Experience Engineers (Support) open bugs in the RHDHSupp project to engage Engineering for assistance \- Captures problem determination process and resolution (Document workaround, link to RHDHBugs for defects or link to Feature Request in RHDHPlan) | RHDHSupp Bug Workflow [RHDH Support Process](https://docs.google.com/document/d/1hd5Acy9y9ZERKY7TBIhsPr1GQqJuCrIETVZUkHAYkPA/edit?tab=t.0) **NEW Status:** Description, Acceptance criteria, Refinement, Story point **Refinement status**: Acceptance Criteria, Story Point, Team, Priority  **Backlog:**  Waiting to be assigned and included in a sprint **IN PROGRESS status:** Response needed from Engineering **Review status:** Waiting for feedback from Support/Customer **Closed status:**  Response provided | [RHDHSupp](https://issues.redhat.com/secure/Dashboard.jspa?selectPageId=12370991) |
